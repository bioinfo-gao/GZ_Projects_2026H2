## 0. 一句话：这个分析在干嘛

把**打碎的宏基因组测序 reads 从头拼成 contigs（组装），再把 contigs 按"来自同一个基因组"归堆（分箱 binning），从而重建出一个个近完整的微生物基因组草图（MAG）**，最后判断每个 MAG 是谁（GTDB 分类）、编码什么基因（注释）。

- 它回答的是："这个样本里能拼出哪些基因组？它们是什么物种？——包括**数据库里可能没有的、没被培养过的新菌**。"
- 它**不**回答"各已知物种丰度多少"——那是 assembly-free 的活（`/taxnom` 的 Kraken2/MetaPhlAn）。mag 5.x 已经把 reads 层分类整个移除了，专心做"组装→基因组重建"。

**先想清楚再动手**：只想知道群落组成/丰度 → 用 `/taxnom`（快、省）。要基因组级证据/发现新物种 → 才用本流程（慢、吃内存）。

## 1. 流程每一步做什么、为什么

| 步骤 | 工具 | 在做什么 | 为什么需要 |
| :--- | :---: | :--- | :--- |
| ① 质控 QC | fastp | 去接头、去低质量碱基 | 脏 reads 会让组装碎、引入错误 contig |
| ② 去宿主 | Bowtie2 | 把比对到宿主(人/鼠)基因组的 reads 扔掉 | 宿主样本(肠道/组织)里大量是宿主 DNA，不去掉会污染组装、浪费算力 |
| ③ 组装 | **MEGAHIT / metaSPAdes** | 把短 reads 拼成长 contigs | 分箱和基因预测都得先有 contigs；这是**整个流程最吃内存的一步** |
| ④ 组装评估 | QUAST | 算 N50、总长、最长 contig | 判断组装质量好坏，碎不碎 |
| ⑤ 分箱 | MetaBAT2/MaxBin2/SemiBin2… | 按覆盖度+四联体频率把 contigs 归成一个个"箱"(每箱≈一个基因组) | 组装出的 contigs 是混在一起的，分箱才能得到单个基因组 |
| ⑥ bin 质控 | **CheckM2** / BUSCO | 评估每个 bin 的**完整度**和**污染度** | 区分"近完整干净基因组"和"垃圾/嵌合 bin"，决定哪些能用 |
| ⑦ bin 分类 | GTDB-Tk | 把合格的 bin 放到 GTDB 系统发育树上定种 | 知道每个 MAG 是什么物种（含新类群） |
| ⑧ 基因预测/注释 | Prodigal + Prokka | 预测基因、注释功能 | 拿到基因组后看它编码什么 |
| ⑨ 汇总 | MultiQC | 把各步 QC 汇成一张报告 | 一眼看全流程健康度 |

## 2. 关键参数为什么这么选

### 🔴 最重要：复杂样本 `--skip_spades`，只用 MEGAHIT

**这是这台机器上最关键的一个决定，核心是内存。**

nf-core/mag 默认**同时**跑两个组装器：MEGAHIT 和 metaSPAdes。它们的差别在内存：

| | MEGAHIT | metaSPAdes |
| :--- | :---: | :---: |
| 算法 | succinct de Bruijn graph（**内存压缩**） | 多 k-mer + 图纠错（**内存换质量**） |
| 单个肠道样本内存 | ~10–40 GB | ~40–120+ GB |
| 高复杂样本(土壤/深海) | 几十 GB | **200 GB–1 TB** ← 会把本机 125 GB 撑爆(OOM) |
| 组装质量 | 好、够用 | 通常更连续一点点 |
| 速度 | 快 | 慢 2–5 倍 |

**原理**：metaSPAdes 把整张 de Bruijn 图 + 多个 k-mer 层 + 纠错结构都放内存里，样本越复杂（物种越多、越深），图越大、内存越涨，没有上限保护；而 MEGAHIT 用"简洁图"数据结构，把同样的信息压到极小的内存里。所以**样本一复杂，metaSPAdes 先爆**，MEGAHIT 还能扛。

> ⚠️ 关键认知：把 `local_resources.config` 的 `resourceLimits` 设成 96 GB **并不能让 metaSPAdes 变小**——那只是"限额"。metaSPAdes 拿到 `-m 96` 后，一旦真的需要更多就**直接报错退出**，不是自动省着用。所以复杂样本的正确做法是**从一开始就 `--skip_spades`，不给它机会爆**。

**什么时候可以用 metaSPAdes？** 样本明确是**低复杂度**（分离株/共培养、简单群落、mock）、且你想要稍微更好的组装连续性时，可以开着 SPAdes。**默认对复杂/环境样本一律只用 MEGAHIT。**

### 其他参数

| 参数 | 为什么 |
| :--- | :--- |
| `-r 5.4.2` 锁版本 | 结果可复现；也避免升级引入未知行为 |
| `-profile singularity` + 共享镜像缓存 | 每个工具在容器里跑，不污染 conda 环境；共享缓存不重复下载几十个镜像 |
| `process.resourceLimits`（不是 `--max_cpus`） | 5.4.2 现代模板只认这个；把总量压在 24 核/96 GB，守住"≤28 核/56 线程、留内存给系统"的底线 |
| **只留 3 个 binner**（metabat2+maxbin2+semibin2，`--skip_concoct --skip_comebin --skip_metabinner`） | 6 个全开又慢又吃资源，这 3 个 + DAS Tool 精炼已能拿到高质量 bin；CONCOCT/COMEBin/MetaBinner 增益小、耗时大 |
| `--run_checkm2` 做主质控 | CheckM2 基于 DIAMOND，又快又轻(~10–15 GB)，完整度/污染度比 BUSCO 更贴合原核 |
| GTDB-Tk **用默认 split-tree，绝不 `--gtdbtk_use_full_tree`** | split-tree 的 pplacer 步 ~55–70 GB(本机扛得住)；full-tree 要 **~320 GB**，本机 125 GB 直接跑不了 |
| **跳过 CAT**（不给 `--cat_db`） | CAT 的 nr 库要 100–200 GB 内存 + 巨大磁盘，是内存黑洞；GTDB-Tk 已给 bin 分类，一般不需要 CAT |
| `--gtdbtk_min_completeness 50` | 只对够完整的 bin 做分类，省算力、避免给垃圾 bin 贴错标签 |
| **`-work-dir /mnt/ex_8T_SSD/...`** | `/Work_bio` 是**机械盘**且快满(898 GB)；`/mnt/ex_8T_SSD` 是**固态盘**、3.9 TB、可随时删。组装中间文件海量且 I/O 密集，放 SSD 又快又不挤爆 HDD。最终 `outdir`(体量小)留项目目录 |

## 3. 三类样本怎么分、区别到底在哪

下一节按"宿主相关 / 中等复杂环境 / 高复杂环境"三档判断可行性。这三档到底是按什么分的？**核心只有一个轴：微生物群落的复杂度**——具体是三个子指标叠加，它们直接决定组装时 de Bruijn 图有多大、多纠缠，从而决定内存和能不能拼出好 MAG：

1. **物种丰富度**（有多少种不同的基因组）——种越多，图越大。
2. **均匀度 / 有无优势菌**（是少数几种占主导，还是几千种势均力敌）——越均匀越难，因为没有高覆盖的"主干"可依托。
3. **株系微多样性**（同一物种有多少个相近菌株混在一起）——株系越多，组装图越纠缠、越碎，这是土壤最要命的地方。

> 一句话：**种越多、越均匀、株系越杂 = 越复杂 = 组装越吃内存、MAG 越难拼。** 另有一个次要轴——**是否需要去宿主**（宿主相关样本要，环境样本一般不要）。

### 三档的典型例子与特征

| 档位 | 典型样本举例 | 物种丰富度 | 优势菌/均匀度 | 株系微多样性 | 去宿主 | 组装难度 & 本机 |
| :--- | :--- | :---: | :---: | :---: | :---: | :--- |
| **宿主相关**（低–中复杂） | 人/动物**粪便·肠道**、口腔、皮肤、阴道、痰/呼吸道、临床拭子/伤口、发酵食品(酸奶/泡菜/kefir)、昆虫肠道、**mock 群落/分离株共培养** | 中(几十~几百种) | **常有优势菌主导**(好事，易拼) | 低–中 | **需要** | ✅ 内存友好，本机稳跑 |
| **中等复杂环境** | **厌氧消化器/沼气池**、**活性污泥/污水处理**、工业**发酵罐/生物反应器**、**淡水**(湖/河表层)、饮用水管网**生物膜**、**海水表层**、**富集培养**、**堆肥**、**反刍动物瘤胃(牛/羊瘤胃液)**、建成环境(医院/地铁表面) | 较高(几百~上千种) | 有一定优势类群，但比肠道均匀 | 中 | 视来源(瘤胃需去宿主，环境样不需) | ⚠️ metaSPAdes 可能吃 100+ GB → **建议只用 MEGAHIT** |
| **高复杂环境** | **土壤**(农田/森林土)、**沉积物**(海底/湖底)、**深海**、腐殖质、部分开阔**海水**、原始未富集环境样 | **极高(数千~上万种)** | **高度均匀、几乎无优势菌**(最难) | **极高(海量近缘株系)** | 不需要 | ❌ 图巨大且极度纠缠，metaSPAdes 需 200 GB–1 TB，本机内存不够 |

**粪便归哪一档？** 归**第一档（宿主相关）**。粪便虽然是排出体外的样本，但它承载的就是**肠道菌群**——物种中等丰富、有拟杆菌/厚壁菌等**优势类群**、组装友好；且常混有宿主肠上皮细胞和食物 DNA，所以**需要去宿主**。它是本流程最典型、最好跑的样本类型，本机稳跑。

**为什么中等和高复杂差别这么大**：肠道靠"几个优势菌高覆盖"能拼出主干；到了土壤，成千上万个物种丰度都很低且彼此接近，外加海量近缘株系，de Bruijn 图既大又缠成一团——metaSPAdes 内存指数级上涨、组装严重碎片化、低丰度菌根本达不到能拼出 MAG 的覆盖度。所以**土壤是公认最难的宏基因组**；中等复杂(污泥/沼气池/水体)介于两者之间：比肠道杂，但还没到土壤那种"什么都拼不动"的地步，用 MEGAHIT + 足够深度通常能拿到一批 MAG。

> ⚠️ 关键认知：**分档看的是群落复杂度，不是"是不是宿主样本"。** 同样是宿主相关采样，**粪便/肠道**有优势菌、好拼，落**第一档**；而**反刍动物瘤胃、白蚁肠道**群落极其丰富，复杂度接近中等环境样本，已列入**中等档**——别按普通肠道估内存。反过来，环境样也不都难：**酸矿废水(AMD)** 物种极少，比肠道还好拼。**先看复杂度，再套档位。**

## 4. 本机能跑什么样本（重点结论）

**机器**：AMD Threadripper 2990WX，32 核/64 线程，**125 GB 内存**。CPU 富余，**内存是唯一硬墙**（磁盘已用 SSD 解决）。

| 样本类型 | 常规数据量 | 能否跑 | 说明 |
| :--- | :---: | :---: | :--- |
| 宿主相关(肠道/口腔/皮肤/临床/mock/分离株群落) | 5–10 Gbp/样(≈3–8 GB gz) | ✅ **稳跑** | 主力场景，裁剪配置下没问题 |
| 中等复杂环境(污泥/沼气池/发酵罐/淡水/生物膜) | 8–15 Gbp/样 | ⚠️ 可行 | 建议 `--skip_spades` 只用 MEGAHIT |
| 高复杂环境(**土壤**、沉积物、深海) | 15–30+ Gbp/样 | ❌ 风险大 | metaSPAdes 需 200 GB–1 TB；对策：MEGAHIT-only + 抽样，或换大内存机 |

**耗时（单个常规肠道样本）**：
- 裁剪配置（MEGAHIT + 3 binner + CheckM2 + GTDB-Tk、跳过 CAT）：约 **3–6 小时**。
- 默认全套（双组装器 + 6 binner + BUSCO + GTDB-Tk）：约 **8–20 小时**。
- **几个样本(3–6)**：分箱/质控能并行一部分，但 metaSPAdes 和 GTDB-Tk 这类吃内存的步骤会在内存上限下串行 → 现实 **1–3 天**。

**各步内存/时间速查（单肠道样本）**：

| 阶段 | 内存峰值 | 耗时 |
| :--- | :---: | :---: |
| fastp QC | 2–4 GB | 5–15 min |
| 去宿主(Bowtie2) | 10–20 GB | 15–40 min |
| MEGAHIT | 10–40 GB | 0.5–2 h |
| metaSPAdes（若开） | **40–120+ GB** | 2–8 h |
| 分箱(每 binner) | 5–20 GB | 合计 1–4 h |
| CheckM2 / BUSCO | 10–15 GB | 10–40 min |
| GTDB-Tk(split-tree) | **55–70 GB** | 0.5–1.5 h |
| CAT（不建议开） | 100–200 GB | — |

## 5. 上云跑（AWS）：时间与总费用估算

如果本机内存不够（尤其土壤）或想并行多样本，可以租 AWS 按需算。前提按你的用法设定：**跑完就把结果下载到本地、不在云端长期存储**（因此没有月度 S3 存储费，只有跑的当下的 EBS 临时盘 + 下载流量）。

**费用由四块组成**（us-east-1 按需价，约数）：

1. **计算（EC2）——大头**。组装吃内存，选**内存优化**机型：
   - 宿主/中等复杂：`r6i.8xlarge`(32 vCPU/256 GB, ~$2.02/h) 或 `r6i.16xlarge`(64 vCPU/512 GB, ~$4.03/h)
   - 高复杂(土壤)：`x2iedn.8xlarge`(32 vCPU/1 TB, ~$6.67/h) ~ `x2iedn.16xlarge`(64 vCPU/2 TB, ~$13.34/h)
2. **临时磁盘（EBS gp3）**：跑的时候放 work/，~$0.08/GB·月，按小时折算。单样本 0.5–4 TB 盘跑 1–3 天 ≈ **$1–50**，是小头。
3. **网络流量**：
   - **上传原始数据到 AWS：免费**（AWS 入站流量不收费）。
   - **下载结果到本地：收费**，约 **$0.09/GB**（超出每月免费 100 GB 的部分；MAG/bin/注释/报告体量比原始数据小得多，常规几~几十 GB，几个样往往落在免费额度内）。
4. **长期存储：$0**（你不存，跑完即下即删）。

### 各档估算（单样本，按需价，含一切费用）

| 档位 | 推荐机型 | 单样本墙钟 | 计算费 | EBS | 下载流量 | **单样本总计** |
| :--- | :--- | :---: | :---: | :---: | :---: | :---: |
| **宿主相关**(肠道/粪便, 5–10 Gbp) | r6i.8xlarge (256 GB) | 4–8 h | ~$8–16 | ~$1 | $0–1.5 | **≈ $10–18** |
| **中等复杂**(污泥/沼气池/瘤胃, 8–15 Gbp) | r6i.16xlarge (512 GB)，或 256 GB 只跑 MEGAHIT | 8–16 h | $20–65 | ~$3 | $0–3 | **≈ $25–70** |
| **高复杂**(土壤/深海, 15–30+ Gbp) | x2iedn.8xlarge~16xlarge (1–2 TB) | 24–72 h | $160–960 | $20–50 | $3–7 | **≈ $200–750** |

> 高复杂档变数最大：metaSPAdes 跑深土壤可达数天、需 1–2 TB 内存；下限对应"MEGAHIT-only + 1 TB 机型"，上限对应"全 metaSPAdes + 2 TB 机型跑满 3 天"。

### 省钱与提速要点

- **Spot 竞价实例可省 ~65–70% 计算费**——把上表计算费 ×0.3~0.35。但 Spot 会被回收，多天的组装有中断风险；配合 nf-core 的 `-resume`（work/ 放持久 EBS）可续跑，适合能容忍重试的场景。土壤档用 Spot 最划算（$200–750 → ~$70–260）。

> **📌 Spot 是什么、为什么便宜（机制说明）**
>
> AWS 的物理服务器永远有一大批**闲置产能**（没被按需/预留客户租走的机器）。与其让它空转，AWS 把这批闲置产能打折甩卖，就叫 **Spot 竞价实例**——同一台机器、同样的性能，价格常低至按需价的 **1/3 左右（省 60–90%）**。本质是"临期机票/尾房"式的甩卖：闲置产能卖不掉就是纯损失，打折卖掉也比 $0 强。
>
> **为什么能这么便宜？因为它不保证可用性——这正是折扣的来源。** 代价是：当 AWS 需要把这批产能收回给按需/预留客户，或该机型该可用区的 Spot 需求上涨时，它会**提前 2 分钟通知、然后回收你的实例**。你用**可靠性换价格**；按需之所以贵，是因为它保证"你不主动停，机器就一直是你的"。
>
> **价格怎么定**：如今 Spot 价按各机型×可用区的供需**平缓浮动**（2017 年后 AWS 已废除早期那种剧烈竞拍，价格变化平滑）。你付当前 Spot 价，可设一个上限（默认=按需价）；一旦 Spot 价超过你的上限、或产能被收回，就触发中断。
>
> **对组装意味着什么**：跑得越久（土壤那种数天的 metaSPAdes），"至少被回收一次"的概率越高。缓解办法——① work/ 放**持久 EBS** + Nextflow `-resume`，被回收的任务只是重跑那一个 task，已完成的走缓存不重算；② 用 **AWS Batch + Spot**，它会自动把中断的任务在新的 Spot 产能上重试。做好断点续跑后，Spot 中断的代价只是"重跑一个步骤"，不是"整轮白跑"。宿主/中等档跑几小时，中断概率本就低，用 Spot 基本无痛；**唯一要当心的是"单台大机器裸跑一个不可续的长 metaSPAdes"——那种一旦被回收就整个组装作废，宁可用按需（下面解释为什么 `-resume` 也救不了它）。**

> **📌 为什么 `nextflow -resume` 救不了裸跑的长 metaSPAdes**
>
> 关键在 **`-resume` 的恢复颗粒度是"整个已完成的 task"，不是 task 内部的进度**：
>
> 1. **只有跑完才进缓存**。Nextflow 只在一个 task **成功退出（exit 0）、输出文件齐全**时，才把它记进缓存（`.nextflow/` cache）。被中途杀死的 task，work 目录里只剩残缺的中间文件，Nextflow **不信任、不复用**，下次 resume 时把这个 task **整个从头重跑**。
> 2. **一次 metaSPAdes 组装 = 一个 task**。假设它要跑 20 小时，第 18 小时机器被 Spot 回收——这个 task **从没"完成"过 → 缓存里没有它 → resume 时从第 0 分钟重新拼**。你能省下的只是它**前面**那些已完成的小步骤（QC、去宿主），而在"被这一坨巨型组装主导"的运行里，这些只占零头。换句话说：**`-resume` 保护的是 task 与 task *之间*，而不是一个长 task *内部*。**
> 3. **SPAdes 其实有自己的内部断点**（`spades.py --continue`），但 **Nextflow / nf-core 默认不调用它**——每次都是发起一条干净的 spades 命令，所以那个"组装到一半"的内部检查点用不上（除非自己改 SPADES 模块去侦测并续跑，属高级定制）。
> 4. **还有个前提常被忽略**：连 task *之间*的 resume 也要求 work/ 和 `.nextflow` 缓存在中断后**存活**。若把 work/ 放在随实例销毁的 **instance-store 临时盘**，实例一 terminate 全没了，连已完成的 task 都救不回——所以 work/ 必须放**持久 EBS**。但即便如此，也改变不了第 2 点：那个**没跑完的巨型组装任务本身**依然没有可复用的缓存。
>
> **所以结论**：`-resume` 对"很多个中小任务、丢了一部分"极有效；但对"**一个又长又不可切分的重型任务**"无能为力——它要么整段完成、要么整段作废。AWS Batch 的自动重试也一样（重试 = 把该 task 从头再跑）。这个长 metaSPAdes 想又省钱又抗中断，只有三条路：**① 用按需/预留保证不被回收；② 换 MEGAHIT（快、内存小，几小时内跑完，落进"中断概率低"的区间）；③ 自定义 SPADES 模块让它用 `--continue` 续跑**。默认配置下，前两条才是现实选择。
- **云的最大优势是并行**：本机上"几个样本"要串行排队（1–3 天）；云上可同时开 N 台 / 用 **AWS Batch**（nf-core 的 `-profile awsbatch`）自动扩缩，**3 个样本的墙钟 ≈ 1 个样本**，只是花 3 倍钱。所以云更适合"样本多、要快出结果"。
- **区域选 us-east-1** 最便宜；下载量大时留意每月 100 GB 免费流量额度。
- 关掉即停：跑完 `terminate` 实例、删 EBS，**不留任何长期费用**。

## 6. 一句话决策指南

- **想要丰度/多样性** → 别用本流程，用 `/taxnom`。
- **要重建基因组(MAG)**，且样本是**宿主相关/中等复杂、常规深度** → 本机能跑，用裁剪配置（MEGAHIT-only、3 binner、CheckM2、GTDB-Tk split-tree、跳 CAT、work 放 SSD），单样本 3–6 小时。
- **样本是深测序的高复杂环境(尤其土壤)** → 本机内存不够，先 MEGAHIT-only + 抽样试；真要全量/高质量，**上 AWS**（见第 5 节：土壤单样约 $200–750，Spot 可降到 ~$70–260）。
- **样本多、要快出结果** → 上 AWS 并行（3 样墙钟≈1 样，只多花钱不多花时间）；宿主相关样本上云极便宜（单样十几美元）。
- 记住那条铁律：**复杂就 `--skip_spades`，先保证跑完，再谈组装质量。**
