## 0. 一句话

**监控不是运维琐事，它是损失的乘数。** 同一个 bug，有监控值 1 小时，没监控值 6 小时。
本方案把「出事到被发现」的延迟从 **11 小时** 压到 **最坏 30 分钟**，且**整条链没有一环依赖 agent（Claude）在场**。

---

## 1. 事故：为什么需要这套东西

2026-07-16，proj16 的 sarek 因一个提速 flag 触发单线程 `GATK4_ESTIMATELIBRARYCOMPLEXITY` 空转。

| 时刻 | 事件 |
| :--- | :--- |
| `07-15 23:50` | 带 Spark flag 重启 |
| `07-15 23:53` | Phase-1 监控打出 `monitor done` 后**自行退出** |
| `07-16 04:30` | ELC 开始空转（load 32 → 个位数，~53 核闲置） |
| `07-16 10:52` | **人工**发现并回退 |

- **监控空窗 11.0 h**；ELC 空转 **6.4 h** 无人察觉。
- 按 10–20 min 巡检，**最晚 05:00 就该发现** → 多烧约 6 小时。

**根因有四层，每层都是一个独立的洞：**

1. **Phase-1 通过被当成「监控已完成」** —— Phase-2 从未启动。
2. **监控进程跑在 Bash `run_in_background` 里** —— 与 agent 会话同生共死。
3. **旧看门狗的规则抓不到这类故障** —— 见 §3。
4. **告警没有投递路径** —— 见 §4。

## 2. 架构：四层，每层都不依赖 agent

```
  ┌─ 作业层    tmux <proj>_wgs        ← 长任务本体（sarek 等）
  │
  ├─ 监控层    tmux <proj>_watch      ← 看门狗，永不退出，10 min 巡检
  │              nextflow_watchdog.sh (WD_PERSIST=1)
  │
  ├─ 守护层    crontab 每 30 min      ← 看门狗死了？复活它 + 发信告知
  │              wd_guard.sh          ← **监控本身也需要被监控**
  │
  └─ 投递层    wd_sendmail.py         ← 脚本自带 SMTP，唯一覆盖夜间的渠道
                 + 定向写用户 tty
                 + (可选) agent Monitor → IDE 弹窗
```

**脚本位置（绝对路径，勿移动）**：

| 脚本 | 路径 | 作用 |
| :--- | :--- | :--- |
| 看门狗 | `facilities/Server/nextflow_watchdog.sh` | 巡检 + 告警 |
| 发信器 | `facilities/Server/wd_sendmail.py` | 自主 SMTP 发信（不含密码） |
| 守护 | `facilities/Server/wd_guard.sh` | 复活看门狗 |

> ⚠ **这三个脚本被 6 处文档 + `/corun` skill + crontab 绝对路径引用。移动它们需同步改 7 个地方，
> 漏一个就静默失效** —— 故意留在 `Server/` 根目录，本文档只放在子目录里。

## 3. 五条告警规则与**标定理由**（最重要的一节）

| 退出码 | 规则 | 触发条件 | 抓什么 |
| :--- | :--- | :--- | :--- |
| 10 | `SESSION_END` | tmux 会话消失 | 作业结束/崩溃 |
| 11 | `LOW_MEM` | available < 2.5 GB 连续 2 次 | OOM 风险 |
| 12 | `HEARTBEAT` | 健康心跳 | 存活确认 |
| **13** | `STALLED` | `.nextflow.log` 停滞 > 45 min | **仅「nextflow 本体死了」** |
| **14** | `LOW_CPU_UTILIZATION` | load < 10 持续 **4 h** | 核在空转 |
| **15** | `LONG_TASK` | 单 task 运行 > **5 h** | **最有用：直接点名哪个 task 在磨** |

### ⚠⚠ 第一版 13/14 两条规则**都标定错了**（实测修正，教训）

**(1) `.nextflow.log` 的 mtime 是【存活心跳】，不是【进度信号】。**
实测 ELC 空转的 6.4 h 里，nflog **每小时稳定写 24 行**（约每 2.5 min 一次），mtime **从未变旧**。
→ `STALLED(45min)` 在真停滞时**永远不会触发**。它的真实作用域**仅限「nextflow 进程死了/挂了」**。
（第一版曾在教训文档里错写「看门狗只会记下 nflog+300m」—— nflog 根本到不了 +300m。）

**(2) 低 load ≠ 故障。**
常规 `GATK4_MARKDUPLICATES` 近单线程、**合法**运行 ~3.5 h/样本，此时 load 仅 ~4-6。
第一版把 LOW_CPU 阈值设 30 min → **每个正常 markdup 阶段都会狼来了**。
**告警一旦狼来了，就等于没有告警。** 故阈值必须 **长于最长的合法单线程阶段**（→ 4 h）。

**(3) 唯一能区分「ELC 空转」与「合法 markdup」的是「单 task 已运行时长」**（前者 6.4 h 无界、后者 ~3.5 h 有界）
→ 这就是 rule 15 存在的理由。实现：扫 `work/` 下有 `.command.begin` 但无 `.exitcode` 的目录算时长。
**注意它不会把 nextflow 主进程（可运行 5h+）误判为超时**，因为它只看 task 目录。

### 判断「低 load 是否正常」的通用判据

> **load 本身永远不可解读，它必须和「当前在跑哪个 task」配对才成为信号。**

1. **哪个 task 在跑** → `grep -a "Submitted process" .nextflow.log | tail -3`
2. **它的预期 CPU 画像与耗时** → 查项目 plan §7 / `/wgs` skill / 历史 `execution_trace_*.txt`
3. **已跑时长 vs 预期** → 在预期内=正常；超出=查

实例对照（同样是低 load，一个正常一个是事故）：

| | 当前 task | 已跑 | 预期 | 判定 |
| :--- | :--- | :---: | :---: | :---: |
| 正常 | `GATK4_MARKDUPLICATES` | 1 h | ~3.5 h（单线程，设计如此） | ✅ |
| 事故 | `GATK4_ESTIMATELIBRARYCOMPLEXITY` | 6.4 h | **本不该存在**（Spark flag 引入） | 🚨 |

## 4. 投递：**只写日志文件的告警 = 日记，不是告警**

监控做对了、告警也触发了，**但没人读到，等于没监控**。
第一版看门狗把告警写进 `logs/watchdog_ALERTS.log` 就完事 —— 那文件只有 agent 去读才会被读到，
而 agent 只在用户开口问时才读 → **「如果用户不问，永远不会发现」**，与「监控 3 分钟就退出」是同一个洞。

| 渠道 | 依赖 | 夜间（agent 不在）有效？ |
| :--- | :--- | :---: |
| 写 `$ALERT_LOG` | 无 | ❌ 没人读 |
| **`wd_sendmail.py` 自主发信** | app password | ✅ **唯一真正覆盖夜间的** |
| 定向写用户 tty | 用户开着终端 | ⚠️ 睡觉时无效 |
| agent Monitor → IDE 弹窗 | agent 会话在线 | ❌ |
| agent 调 Gmail MCP | agent 会话在线 | ❌ |

- **bash 脚本【调不了 MCP】** —— MCP 只有 agent 能调。想让 tmux 里的看门狗自己发信，
  必须给它独立 SMTP 凭据，**没有第三条路**。
- **不要用 `wall`** —— 它广播给机器上所有用户。要定向就只写运行者自己的 tty。
- **Positron / VS Code 弹窗：无路径。** 外部脚本无法调用 IDE 通知 API（需装扩展）。

### ⚠⚠ tty 枚举必须用 `ps`，**绝不能用 `who`**（实测踩到）

`who` 读 utmp **登录会话**记录 —— **IDE 集成终端不注册 utmp**。实测本机：

| 枚举方式 | 见到的 pts | 含用户真正在用的终端？ |
| :--- | :---: | :---: |
| `who` | 5（全是 tmux pane） | ❌ **漏掉 pts/1** |
| `ps -eo tty -u $USER` | 15 | ✅ |

用户当时正在 Positron 的 **pts/1** 工作，`who` 里没有它 → **告警发不到人眼前，日志却显示「已发送到 5 个终端 ✅」**。

```bash
# 正确写法
for t in $(ps -eo tty -u "$USER" | grep -oE 'pts/[0-9]+' | sort -u); do
  [ -w "/dev/$t" ] || continue
  printf '...' > "/dev/$t"
done
```

## 5. 凭据安全（照做，别简化）

```
~/.config/nextflow_watchdog/smtp.env     chmod 600，目录 chmod 700
```

| 规则 | 理由 |
| :--- | :--- |
| **必须放在 git repo 之外** | `projects_2026H2` **是 git repo** —— 凭据放进去迟早被提交 |
| 用 **app password**，不是主密码 | 可在 Google 账号内**独立吊销**，不影响 Claude Code 的 Gmail 功能 |
| **发信脚本本身不含密码**，从文件读 | 故 `wd_sendmail.py` 可安全入库 |
| 异常信息不回显密码 | 见脚本内 `except` 分支 |
| 装好后核查 repo 无残留 | `git -C <repo> grep -l "<password>"` |

> **为什么不复用 Claude Code 的 Gmail MCP 凭据**：那是把凭据从其被授权的路径挪到独立脚本，
> 安全策略会拦（且拦得对）。**专用 app password 反而是更好的设计**：可独立吊销、不与
> `.claude.json` 耦合（后者一旦轮换，看门狗会**静默失效**）。

## 6. 部署到一个新项目（复制即用）

设新项目会话名前缀为 `<P>`（如 `pan`），项目目录 `$PROJ`：

**① 建项目内薄封装** `scripts/10_watchdog.sh`（勿复制看门狗本体，直接调共享实现）：

```bash
#!/usr/bin/env bash
set -uo pipefail
PROJ=/home/gao/projects_2026H2/<你的项目>
exec env \
  WD_PERSIST=1 WD_POLL=600 \
  WD_LOG="$PROJ/logs/watchdog.log" \
  WD_ALERT_LOG="$PROJ/logs/watchdog_ALERTS.log" \
  WD_STALE_MIN=45 WD_LOW_LOAD=10 WD_LOW_LOAD_MIN=240 WD_MAX_TASK_H=5 \
  bash /home/gao/projects_2026H2/facilities/Server/nextflow_watchdog.sh \
  "<P>_wgs:$PROJ:$PROJ/logs/run.log"
```

**② 起作业 + 立刻起看门狗**（两条一起，别拆开）：

```bash
tmux new-session -d -s <P>_wgs   "cd $PROJ && <nextflow run ...> 2>&1 | tee -a logs/run.log"
tmux new-session -d -s <P>_watch "bash $PROJ/scripts/10_watchdog.sh"
```

**③ 装守护**（每 30 min；`9,39` 而非 `0,30` 以避开整点拥挤）：

```bash
crontab -e   # 追加
9,39 * * * * /home/gao/projects_2026H2/facilities/Server/wd_guard.sh <P>_wgs <P>_watch 'bash /home/gao/projects_2026H2/<项目>/scripts/10_watchdog.sh' >/dev/null 2>&1
```

**④ 验证（必做，见 §7）。**

**⑤ 项目结束后**：`crontab -e` 删掉该行（否则作业没了，guard 每 30 min 空跑一次——虽会安静退出，但属残留）。

### 频率标定

> **目标是把「发现延迟」压到有界，不是追求秒级复活**（user directive 2026-07-16）。
> 30 min 足矣 —— 最坏丢 30 min 监控，而事故当晚丢的是 11 h。
> 更密（如 13 min）属过度设计：守护本身耗资源，收益只是 30→13 min，
> 与「10+ 小时 → 30 分钟」的数量级改进相比可忽略。

## 7. 验证：**「启动了」≠「会告警」**

**这是本方案最重要的纪律。** 事故的同型错误就是把「启动」当「生效」——所以监控本身**必须实测**。

```bash
# ① LONG_TASK 会触发（造一个 7h 的假 task）
mkdir -p /tmp/t/proj/work/zz/aa && touch -d "7 hours ago" /tmp/t/proj/work/zz/aa/.command.begin
tmux new-session -d -s fake 'sleep 300'; touch /tmp/t/proj/.nextflow.log; : > /tmp/t/proj/run.log
WD_LOG=/tmp/t/w.log WD_ALERT_LOG=/tmp/t/a.log WD_POLL=1 WD_STALE_MIN=0 WD_LOW_LOAD=0 WD_MAX_TASK_H=5 \
  bash facilities/Server/nextflow_watchdog.sh fake:/tmp/t/proj:/tmp/t/proj/run.log; echo "exit=$?"   # 期望 15

# ② 反例：3h 的 task 不该误报（markdup 合法）
touch -d "3 hours ago" /tmp/t/proj/work/zz/aa/.command.begin; rm -f /tmp/.wd_longtask_*
#   → 期望 exit 12（健康心跳），且 a.log 无新告警

# ③ 守护能复活看门狗，且【在 cron 环境下】也能（关键：手动跑通 ≠ cron 跑通）
tmux kill-session -t <P>_watch
env -i HOME=/home/gao PATH=/usr/bin:/bin SHELL=/bin/sh /bin/sh -c \
  "facilities/Server/wd_guard.sh <P>_wgs <P>_watch 'bash <项目>/scripts/10_watchdog.sh'"
tmux has-session -t <P>_watch && echo "复活成功"

# ④ 邮件真能发出（不看退出码，看收件箱）
echo "测试" | facilities/Server/wd_sendmail.py "[测试] 投递自测" -
```

**每一条都要看到期望结果，而不是「命令没报错」。**

## 9. 已知局限（如实记录）

1. **`STALLED(13)` 几乎无用** —— nflog 心跳掩盖了停滞，它只能抓「nextflow 进程本身死了」。
   真正的停滞检测靠 `LONG_TASK(15)`。
2. **`LONG_TASK` 的 5 h 阈值是折中** —— markdup 合法跑 3.5 h，故阈值不能更低；
   代价是 ELC 型故障最坏也要 5 h 才报（vs 事故当时的 6.4 h，仅省 1.4 h）。
   **要更早发现，只能引入「每个 process 的预期耗时表」逐一比对**，尚未实现。
3. **tty 投递要求用户开着终端** —— 睡觉时无效。夜间只有邮件可靠。
4. **Positron/VS Code 弹窗无路径**。
5. **guard 依赖 crontab 服务本身存活** —— 若 cron 服务挂了，无人救 guard。（可接受：cron 是系统级服务。）
6. **项目结束需手动摘掉 crontab 行**，否则残留空跑。

## 10. 三条可迁移的通用教训

1. **「Phase 1 通过」只等于「启动没炸」，不等于「监控已完成」。**
   Phase-1 退出的那一刻必须无缝接上 Phase-2，且 **`tmux ls` 里每个长任务会话都要有配对的常驻监控会话**。
2. **监控规则必须覆盖「活着但空转」。**
   绝大多数监控只查「死没死」（进程在否、内存够否），而真实世界里最贵的故障是**「活着但空转」**
   —— 它不触发任何传统告警，可以安静地烧一整夜。
3. **每一层都要问「它坏了谁会知道」。**
   作业坏了 → 看门狗知道；看门狗坏了 → guard 知道；告警发不出去 → **实测过才知道**。
   本项目所有事故（`MT`→`chrMT`、`exit:129` 被读成 `COMPLETED`、cache 31 里只有 1 个 BWAMEM2、
   「已发送 5 个终端 ✅」却没送到人眼前）**全是同一模式：程序不报错，只是没做到该做的事。**
