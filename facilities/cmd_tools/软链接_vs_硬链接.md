软链接（symbolic link / symlink）和硬链接（hard link）都能让"同一份内容"在文件系统里
出现在多个路径下，但底层原理完全不同，适用场景也不同。这份笔记记录一次真实决策
（项目13/14 共享运行记录 `key_operations_log.md` 该怎么链）背后的推理，供以后遇到类似
"这份文件该建两个位置都能看到"的需求时直接参考。

## 1. 核心技术原理

**硬链接（hard link）**：
原理：Linux 文件系统里，真正的数据由 **inode** 承载，"文件名"只是指向某个 inode 的一个
目录项。硬链接就是给同一个 inode **再建一个目录项**——两个路径名本质上是"平等的、指向
同一份数据"，没有"原件 vs 链接"的区分。
比喻：像是同一个仓库有两个门牌号，两个地址都是这个仓库本身，拆掉哪个门牌，仓库还在。

**软链接（symlink）**：
原理：软链接本身是一个独立的小文件，内容就是"一段路径文本"。访问软链接时，操作系统
按这段路径**重新查找**目标文件。它是"指向一个路径"，不是"指向一块数据"。
比喻：像是一张写着地址的纸条，你拿着纸条去按图索骥；纸条不是仓库本身，仓库门牌换了、
纸条就要跟着更新，否则找不到路（"断链"）。

## 2. 优劣对比

| 特性 | 硬链接 | 软链接 |
| :--- | :---: | :---: |
| 是否跨文件系统/挂载点 | ❌ 不行，必须同一分区 | ✅ 可以跨分区/跨盘 |
| 能否链接目录 | ❌ 大多数系统不允许 | ✅ 可以 |
| 目标被删除后 | 数据仍在（inode引用计数没归零） | 变成"断链"，访问报错 |
| 目标被"原地覆盖式"编辑后 | 两边同步更新（同一份数据） | 同步更新（按路径重新解析） |
| 目标被"写新文件再改名覆盖"式编辑后 | **⚠️ 静默失效**，链接指向旧inode | ✅ 正常，按路径找到新内容 |
| `ls -la` 是否能一眼看出"这是个链接" | ❌ 看不出来，和普通文件一样 | ✅ 显示 `-> 目标路径`，一目了然 |
| git 是否理解这个概念 | ❌ 不理解，会当成独立文件存两份 | ✅ 作为特殊blob类型存一份+一条路径 |

## 3. 决定成败的一点：编辑器的"原子保存"会静默弄坏硬链接

这是最容易被忽略、但后果最隐蔽的一条。

大多数编辑器（vim、VSCode、包括 Claude Code 自己的 Edit 工具）**不会**直接在原地覆写
文件内容，而是："写一个新的临时文件 → 用 `rename()` 把它改名覆盖到目标路径"。这么做是
为了保证保存过程是原子的（半路断电/程序崩溃也不会把文件写坏一半）。

问题在于：`rename()` 覆盖的效果，等价于"删掉旧目录项、在同一个文件名下挂一个全新的
inode"。如果之前在别的路径对这个文件建了**硬链接**，那个硬链接指向的还是**旧 inode**
——编辑器保存之后，硬链接那一侧读到的是**保存前的旧内容**，而且**不会有任何报错或警告**，
只会一直悄悄地"过时"下去，直到有人手动对比内容才会发现两边不一致。

软链接完全不受这个影响：它存的是路径，不是inode，每次访问都重新按路径解析，编辑器
不管怎么保存，软链接都能读到最新内容。

## 4. 决定用哪个：看你要的是"数据副本"还是"路径指路"

- **要跨项目/跨目录共享同一份会被反复编辑的文件**（比如两个项目共用的运行日志、配置说明）
  → 用**软链接**。这类文件几乎必然会被编辑器"原子保存"，硬链接迟早会静默失步。
- **要防止某份数据被误删、且以后不会再编辑，只是想在另一个位置也能访问**
  → 硬链接也可以考虑（如去重后的归档文件），但实践中软链接的"看得出link关系+断链会报错"
    这两个可维护性优点，通常也让软链接是更省心的默认选择。
- **要跨磁盘分区/挂载点共享**：硬链接直接不可行，只能软链接。
- **要在 git 仓库里管理**：软链接更安全——硬链接对 git 而言就是普通文件，如果链接的两端
  都被 git 跟踪，会把同一份内容重复存两份，违反"同一份信息不要在仓库里存两次"的原则。

## 5. 真实案例（2026-07-09，projects_2026H2）

项目13（`13_Ellen_knockin_wgs`）和项目14（`14_geneedit_lats12_wgs`）并跑时共用一份运行
记录 `key_operations_log.md`，内容会随两边进展持续更新。决策：
- 权威原件放项目13（`13_Ellen_knockin_wgs/logs/key_operations_log.md`，git 跟踪）
- 项目14那份改成**软链接**指向它：
  ```bash
  ln -s ../../13_Ellen_knockin_wgs/logs/key_operations_log.md \
        /home/gao/projects_2026H2/14_geneedit_lats12_wgs/logs/key_operations_log.md
  ```
- 用相对路径（`../../...`）而非绝对路径：如果以后整个 `projects_2026H2` 父目录被整体
  移动/备份到别处，两个子项目之间的相对位置关系不变，链接依然有效；绝对路径则会失效。

选软链接的理由正是第3节那条：这份日志会被 Claude/编辑器反复追加编辑，如果用硬链接，
只要有一次是"写新文件再rename"式的保存（这是常态，不是特例），链接那一侧就会静默停在
旧版本，且没有任何提示——这个风险不能接受，软链接没有这个问题。
