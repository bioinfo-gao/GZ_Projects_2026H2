## 一、结论一句话

**管线端到端跑通，核心科学结论正确**（RAGH→Rag2 on-target 检出、拷贝数 ~0.60≈杂合单拷贝、阴性对照 0.01）。但试跑暴露 **2 个已修 bug + 3 个正式跑前必须处理的问题**（同源臂遮蔽、高深度 artifact、Manta 病态慢）。**首跑的每一个坑都值回票价。**

## 三、病态慢步骤分析与对策（正式跑必看）

### ① MANTA 在 CRAM 上极慢（~7.5h）**且对整合结合部无效** —— 全量跑直接弃用
- **决定性证据（Manta 跑完后复查）**：Manta 输出 2413 条 SV（212 BND/2069 DEL/42 DUP/90 INS），VCF header 声明了 TG_RAGH/TG_MTTH 两个构建体 contig，但**涉及构建体 contig 的记录 = 0 条**。即 7.5h 换来**零个整合结合部断点**——而轻量嵌合读段法（脚本4A）干净命中 RAGH→Rag2。原因：结合部读段因同源臂低 MAPQ 被 Manta 过滤。
- **现象**：`GenerateSVCandidates --bin-count 1` 整基因组当 1 个 bin 串行；edgeRuntimeLog 显示**每个 edge ~60 秒**（正常毫秒级），已处理 25 万+ edges。
- **根因**：Manta 对每个断点 edge 做 `--enable-remote-read-retrieval`，在 **CRAM** 上要反复解码切片（比 BAM 慢几个数量级）；敲入位点的大量嵌合读段又制造海量断点 edge，雪上加霜。
- **对策（正式跑任选，推荐前两个组合）**：
  1. **降级 Manta 为可选**：嵌合读段法（脚本4A）**不靠 Manta 就已检出 RAGH→Rag2**，Manta 非必需。正式跑 sarek 可先 `--tools tiddit`（去掉 manta），SV 确认用脚本4A + DELLY。
  2. 若仍要 Manta：**先 CRAM→BAM 再单独跑 Manta**（脱离 sarek，用 `manta` 环境），或 `--callRegions` 限定到主染色体+构建体 contig，避免全基因组 remote retrieval。
  3. 绝不在 sarek 里让 Manta 直接吃 CRAM 全基因组跑。

### ② GATK4 MarkDuplicates 慢（3h28m，单线程）
- **对策**：sarek 加 `--use_gatk_spark markduplicates` 启用 Spark 多线程版；或接受它作为固定成本（6 样 ×3 波 ≈ 10h 仅去重）。首选试 Spark，注意 Spark 偶发不稳，跑前小验证。

### ③ FASTQC 32m（可选省时）
- 非必需时 `--skip_tools fastqc`（我们已有 fastp 的 QC），每样省半小时。

## 四、最佳/已验证参数（正式跑直接采用）

**sarek（保留）**：`-r 3.8.1 -profile singularity`，`--aligner bwa-mem2`，`--skip_tools baserecalibrator`，`--fasta <hybrid> --fasta_fai <..> --igenomes_ignore --genome null`。
**sarek（正式跑调整）**：`--tools tiddit`（去 manta，见上）；`--skip_tools fastqc`（可选）；`--use_gatk_spark markduplicates`（可选，需小验证）。
**资源配置**（`local_resources.config`，实测未超限）：executor 48 线程 / 108 GB，`queueSize=2`；bwa-mem2 mem 16 线程/50 GB、index 8 线程/60 GB（实测峰值 61 GB，勿低于此）。

**下游分析（关键参数，实测有效）**：
- **拷贝数唯一比对阈值 `MAPQ ≥ 20`**：把同源臂/交叉比对读段过滤掉，阴性对照从"37% 覆盖假象"降到真实的 0.01。
- **拷贝数只在"人源特异区"算，不用全长 contig**（见第六节，这是最大教训）。
- **基线**：样本自身常染色体（chr1–19）中位深度（本样 = 19.0×），本批无纯 WT 即用此。

## 六、最大科学教训：同源臂遮蔽（正式跑脚本必须改）

**定点打靶构建体含小鼠同源臂（与内源位点序列全同）→ 这些区段在混合参考里重复 → 读段多重比对（MAPQ 0）。** 后果有二，都会误导：

- **拷贝数被低估**：按**全长 TG_RAGH** 平均，MAPQ≥20 下只剩 0.08，"像没敲入"；**只在人源细胞因子盒（TG_RAGH:19692-24759）**算 → 覆盖 99.6%、深度 11.5×、**mapq 59.9（全唯一）→ ratio 0.60 ≈ 杂合单拷贝**。**同一样本，看全长还是看人源区，结论天差地别。**
- **假整合位点**：RAGH 样本在阴性 TG_MTTH 上出现"候选位点"，落在内源 **chr5 Htt**——那是内源小鼠 Htt 读段比对到 MTTH 构建体的**小鼠同源臂 + 人源 HTT（与小鼠 ~85% 同源）**，不是真整合。

**修正方向（写进脚本4/5）**：
1. 为每个构建体定义**人源特异区坐标**，拷贝数/整合信号只在其内计算，**遮蔽同源臂**。
2. `MTTH` 特殊：人源 HTT 与小鼠 Htt ~85% 同源，即便人源区也部分交叉比对 → 需 MAPQ≥20 **且**聚焦最分歧的外显子；解读需格外谨慎。
3. 候选整合位点过滤掉**落在内源直系同源位点**（RAGH→Rag2 是 on-target 保留；MTTH 样本落在 chr5 Htt 是 on-target；但"非本品系"构建体落在其内源同源位点 = 交叉比对噪声，剔除）。

## 十、三项精修的落地与 rationale（2026-07-07，已用 RAGH_153 CRAM 自测通过）

把 §三/§六/§七 的教训落进脚本，不重跑 sarek、只用现有 CRAM 自测。新增数据驱动的
`refs/constructs/construct_regions.tsv`（每个构建体的人源特异区 / 同源臂 / 内源靶点坐标，
CD1A 到位加一行即可）。

### 精修 1 — 同源臂遮蔽：拷贝数只在人源特异区算（脚本5）
- **Rationale**：定点打靶构建体的小鼠同源臂与内源位点序列全同 → 在混合参考里重复 →
  臂读段多重比对(MAPQ0) → MAPQ≥20 过滤后为 0。若按**全长 contig** 平均，会把大段 0 深度
  的臂算进去，严重低估拷贝数。
- **自测证据**：TG_RAGH 全长 ratio 0.08（"像没敲入"）vs **人源盒(19692-24759) ratio 0.60
  = 杂合单拷贝**（深度 11.5×、覆盖 99.6%、mapq 59.9 全唯一）。脚本5 现同时输出 human_* 与
  wholecontig_*，后者远低即同源臂稀释的证据。

### 精修 2 — off-target 筛查 + on-target 桥接分离（脚本4）—— 本次最重要的认识
- **试跑揭示的微妙点**：加 MAPQ≥20 后，之前"RAGH→Rag2 检出"也消失了。说明那个信号
  **本质是同源臂共定位**（臂读段在构建体↔内源 Rag2 间多重比对、配偶落回 Rag2），**不是
  真正的跨结合部读段**。定点打靶的 on-target **无法**靠 MAPQ≥20 的唯一读段来"检出"。
- **因此把脚本4 拆成语义不同的两路**：
  - **ON-TARGET 桥接（不加 MAPQ）**：构建体读段配偶落**自身内源靶点**的计数（RAGH_153 = 24 条→Rag2），
    佐证整合在设计靶点。但它不能独立区分"正确整合"vs"臂本就同源"——真正证明 KI 存在+拷贝数的是
    脚本5（人源区 0.60）+ 定点设计。
  - **OFF-TARGET 筛查（MAPQ≥20 唯一比对）**：人源特异读段落到**非**预期位点 = 潜在脱靶随机整合。
    **RAGH_153 = 0 个 → 无脱靶（好结果）。这才是 MAPQ≥20 该回答的问题。**
- **教训**：空的 off-target 结果不是"没检出整合"，而是"没有脱靶"——语义必须在脚本输出里说清楚，
  否则会被误读为失败。

### 精修 2b — "构建体是否存在"门控 + 高深度 artifact 黑名单（脚本4）
- **门控**：某构建体人源区 ratio<0.1（阴性）时跳过其位点——否则内源直系同源位点会冒假整合
  信号（MTTH 在 RAGH 样本假落 chr5 Htt）。自测：TG_MTTH ratio 0.02 → present=no → 正确跳过。
- **artifact 黑名单**：候选位点深度 > 基线×5（RAGH_153 基线 19× → 阈值 95×）即弃，滤掉
  chr1:78.58Mb（548× ≈ 基线 29×）这类塌缩重复假象。

### 精修 3 — 全量跑弃用 Manta（脚本2 `--tools tiddit`）
- **Rationale（硬证据）**：Manta 跑完 7.5h，2413 条 SV 里**涉及构建体 contig 的 = 0 条**
  （见 §三①）——既慢又对这类结合部零检出。整合主力是脚本4 嵌合读段法，SV 确认用 DELLY。
